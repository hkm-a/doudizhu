# 斗地主 (Doudizhu) Constitution

## Core Principles

### I. 游戏规则至上 (Rules First)
斗地主是规则严格的竞技游戏。所有牌型识别、比较、合法性判定必须严格遵循标准斗地主规则。任何规则实现偏差都是最高优先级 bug。规则变更必须有明确的规则来源依据，不可凭直觉修改。

- 54 张牌完整支持（含大小王）
- 牌型：单张、对子、三条、三带一、三带二、顺子、连对、飞机（含带牌）、炸弹、火箭
- 顺子/连对/飞机不含 2 和王（rank ≤ 14）
- 炸弹和火箭的倍数叠加规则
- 春天/反春天检测

### II. 零依赖架构 (Zero Dependencies)
保持纯 HTML/CSS/JS 实现，不引入任何第三方框架或构建工具。代码可直接在浏览器中打开运行，无需 npm install、webpack 或任何构建步骤。

- `game.js`：纯游戏引擎，无 DOM 操作
- `app.js`：UI 控制层，桥接 DOM 与引擎
- `audio.js`：Web Audio API 合成音效，无外部音频文件
- `style.css`：纯 CSS，无预处理器

### III. 引擎与 UI 分离 (Engine-UI Separation)
`game.js` 是纯逻辑引擎，可以在 Node.js 中运行测试，不依赖浏览器 API。所有游戏状态变更通过 `DoudizhuGame` 类的方法完成。`app.js` 负责 DOM 渲染和事件绑定，不包含任何游戏规则逻辑。

- 游戏引擎可独立测试：`node -e "const G=require('./web/game.js'); ..."`
- UI 通过 `refreshUI()` 统一刷新，不直接操作游戏状态
- AI 逻辑在引擎内部，UI 只负责调度（setTimeout）

### IV. AI 可配置性 (Configurable AI)
AI 行为必须可通过设置面板调整，支持不同难度和速度。AI 策略应有明确的层次结构：简单模式（不使用炸弹）、普通模式（标准策略）、困难模式（更积极的炸弹使用）。

- AI 速度：慢速(800ms) / 正常(500ms) / 快速(200ms)
- AI 难度：简单 / 普通 / 困难
- AI 角色感知：地主 vs 农民有不同策略
- 农民 AI 需要配合（让牌给搭档）

### V. 响应式优先 (Mobile-First Responsive)
游戏必须在桌面和移动端都能正常游玩。使用 `clamp()` 实现流式尺寸，通过媒体查询适配不同屏幕。手牌区域支持横向滚动。

- 桌面端：72×100px 卡牌，扇形重叠布局
- 平板：58×82px
- 手机：46×66px
- 最小触摸目标：38px

## 技术约束

### 代码风格
- JavaScript：ES6+，无 class 继承，函数式优先
- CSS：BEM 不强制，但选择器保持简洁
- 命名：中文注释，英文变量名和函数名
- 文件大小：单文件不超过 1200 行（game.js 当前 ~1009 行）

### 性能要求
- 首屏加载 < 500ms（纯静态文件）
- 动画帧率 ≥ 30fps
- AI 响应延迟可配置（200ms-800ms）
- localStorage 读写不影响主线程

### 音效规范
- 全部使用 Web Audio API 合成，无外部文件
- 语音播报使用 SpeechSynthesis API
- 静音状态需持久化
- 每次出牌播报具体牌型和牌面（如"顺子 三四五六七"）

## 开发工作流

### 迭代开发
- 每次迭代产出可玩版本
- 迭代优先级：规则正确性 > AI 智能 > UI/UX > 性能优化
- 每次改动后运行 20 局模拟测试验证（Node.js），目标 100% 完成率

### 测试策略
- 核心测试：Node.js 跑 20 局完整游戏，验证无死循环
- 牌型测试：手动验证每种牌型的识别和合法性
- 边界测试：JQKA2 无效、炸弹/火箭倍数叠加、春天检测

### 文件结构
```
web/
  game.js      — 游戏引擎（规则、AI、状态管理）
  app.js       — UI 控制层（DOM 操作、事件、动画调度）
  audio.js     — 音效合成（Web Audio API + SpeechSynthesis）
  index.html   — 页面结构
  style.css    — 样式（含响应式）
```
